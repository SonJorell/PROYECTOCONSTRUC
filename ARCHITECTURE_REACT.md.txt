# StockSync React
# Blueprint de Arquitectura y Especificación Técnica

> **Versión:** 1.0  
> **Estado:** Documento de Arquitectura Inicial  
> **Framework Frontend:** React 18 + Vite  
> **Lenguaje:** JavaScript (ES2022+)  
> **Estilos:** Tailwind CSS  
> **Backend:** Supabase (Database + Auth + Storage + Realtime)  
> **Tipo de Aplicación:** Web SPA (Single Page Application)

---

# 1. Objetivo del Proyecto

StockSync React es una aplicación web destinada a la administración de inventario, diseñada para ofrecer una interfaz moderna, rápida y escalable para empresas que requieren controlar productos, existencias, movimientos de stock y procesos de carga masiva de información.

La aplicación prioriza:

- Alto rendimiento.
- Escalabilidad.
- Código modular.
- Fácil mantenimiento.
- Integración nativa con Supabase.
- Experiencia de usuario intuitiva.
- Arquitectura preparada para futuras funcionalidades.

# 3. Principios de Arquitectura

La aplicación debe desarrollarse siguiendo los siguientes principios:

## Separación de responsabilidades

Cada componente debe tener una única responsabilidad.

Ejemplos:

- Scanner → únicamente escanear códigos.
- InventoryTable → únicamente mostrar datos.
- ProductModal → únicamente editar productos.

Nunca mezclar lógica de negocio con la presentación.

---

## Código reutilizable

Los componentes deben ser reutilizables.

Evitar:

- Código duplicado
- Funciones repetidas
- Estados innecesarios

---

## Escalabilidad

La estructura debe permitir agregar nuevas funcionalidades sin modificar grandes partes del proyecto.

Ejemplos futuros:

- Multi sucursales
- Usuarios
- Roles
- Auditoría
- Reportes
- Dashboard financiero

---

## Bajo acoplamiento

Los componentes no deben depender entre sí.

Toda comunicación debe realizarse mediante:

- Props
- Context API
- Hooks personalizados

# 4. Configuración del Entorno

## Variables de Entorno

Crear un archivo:

```
.env.local
```

En la raíz del proyecto.

### Variables requeridas

```env
VITE_SUPABASE_URL=https://xxxxxxxx.supabase.co

VITE_SUPABASE_ANON_KEY=xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
```

## Importante

Las variables utilizadas por Vite deben comenzar obligatoriamente con el prefijo:

```
VITE_
```

Estas variables serán consumidas únicamente mediante:

```javascript
import.meta.env
```

Ejemplo:

```javascript
const url = import.meta.env.VITE_SUPABASE_URL;
```

Nunca escribir credenciales directamente dentro del código fuente.

# 7. Responsabilidad de Cada Módulo

## DashboardKPIs

Responsable de mostrar:

- Total de productos
- Stock total
- Productos críticos
- Valor del inventario
- Productos agotados

Debe actualizarse automáticamente al cambiar el inventario.

---

## InventoryTable

Responsable de:

- Mostrar productos
- Ordenar columnas
- Filtrar
- Buscar
- Paginar resultados

No debe contener lógica de conexión con Supabase.

---

## ProductModal

Permite:

- Editar stock
- Editar precio
- Editar categoría
- Editar proveedor
- Guardar cambios

Debe validar los datos antes de enviarlos.

---

## FileDropzone

Responsable de:

- Arrastrar archivos CSV
- Validar formato
- Limpiar datos
- Convertir columnas
- Procesar registros por lotes
- Mostrar progreso
- Mostrar errores

---

## ScannerModal

Permite:

- Abrir cámara
- Escanear códigos de barras
- Buscar automáticamente el producto
- Cerrar cámara correctamente

Debe liberar los recursos de la cámara al cerrar.

---

# 9. Organización de la Lógica

Toda la lógica de negocio debe vivir en:

```
services/
```

Toda la lógica reutilizable debe vivir en:

```
hooks/
```

Toda la lógica auxiliar debe vivir en:

```
utils/
```

Los componentes únicamente deben encargarse del renderizado.

# 11. Objetivos de Rendimiento

La aplicación debe cumplir los siguientes objetivos:

- Tiempo inicial de carga inferior a 2 segundos.
- Renderizado fluido de tablas con miles de registros.
- Consultas paginadas.
- Evitar renderizados innecesarios.
- Uso de React.memo cuando corresponda.
- Uso de useMemo para cálculos pesados.
- Uso de useCallback para funciones reutilizadas.

# 13. Buenas Prácticas

- Nunca duplicar código.
- Mantener componentes pequeños.
- Evitar archivos mayores a 300 líneas cuando sea posible.
- Documentar funciones complejas.
- Validar entradas del usuario.
- Manejar correctamente los errores.
- Utilizar variables de entorno.
- No almacenar claves privadas en el frontend.
- Mantener una estructura consistente en todo el proyecto.
